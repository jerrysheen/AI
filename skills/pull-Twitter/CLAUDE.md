# CLAUDE.md

## 项目默认执行规则

你在本项目中执行任何“实现 / 修复 / 调研 / 自动化”类任务时，必须遵守以下规则：

### 1. 先读进度，再开始
- 开始任何任务前，先检查项目根目录是否存在 `.task_progress.json`
- 如果存在，先读取它，理解：
  - 当前任务目标
  - 已完成阶段
  - 已尝试方案
  - 当前阻塞点
  - 下一步动作
- 如果不存在，则创建一个新的 `.task_progress.json`

### 2. 先规划，再执行
- 不要拿到需求后立刻乱改代码
- 先输出一个简短计划，至少包含：
  - 目标是什么
  - 打算分几个阶段
  - 先做什么验证
  - 什么条件下算完成
- 如果任务涉及多种技术方案，先做最小验证，再决定实现方案

### 3. 严格按阶段推进
默认阶段如下：
1. 理解目标
2. 方案验证
3. 正式实现
4. 本地验证
5. 交付总结

如果任务明显不需要完整五阶段，可以简化，但必须至少包含：
- 方案判断
- 实现
- 验收

### 4. 每个阶段结束都要更新进度
每完成一个阶段，都必须：
- 在终端输出：`[✓] 阶段名`
- 更新 `.task_progress.json`

必须写入：
- 当前阶段
- 已完成步骤
- 当前使用方案
- 尝试失败的方案及原因
- 当前阻塞问题
- 下一步动作
- 当前验收状态

### 5. 不允许跳过验证直接宣称完成
- 任何实现类任务，在没有执行验收命令前，不得声称“完成”
- 若项目中已有测试/脚本/构建命令，优先运行真实命令验证
- 若没有现成验证命令，必须补一个最小可执行验证脚本

### 6. 优先做最小可行验证
- 不要一开始就做过度设计
- 先证明方案可跑通，再整理结构
- 若外部依赖不稳定，先验证依赖、导入、网络访问、最小样例

### 7. 失败时也要可交付
如果任务最终未完全成功，也不能空手结束。
必须交付：
- 已尝试方案列表
- 失败原因矩阵
- 当前环境限制
- 距离成功还缺什么
- 已完成的文件和代码

### 8. 网络 / 外部服务约束
- 只使用环境中已有的代理配置，不自行伪造代理地址
- 如需凭证，只读取已有环境变量或项目配置，不自行编造
- 外部服务失败时，应明确区分：
  - 网络不可达
  - 认证失败
  - 接口变更
  - 解析失败
  - 限流 / 风控
- 不要在外部服务失败时无限重试

### 9. 输出风格要求
- 简洁汇报当前进度
- 明确说明“当前采用什么方案”
- 明确说明“为什么放弃别的方案”
- 贴出真实执行命令和结果摘要
- 不要用 mock 数据冒充成功

### 10. 对实现类任务的默认完成定义
只有满足以下条件，才可算完成：
- 代码已落盘
- 关键逻辑已实现
- 验收命令已运行
- 验收结果满足要求
- README / 使用说明已更新
- `.task_progress.json` 已写入最终状态
