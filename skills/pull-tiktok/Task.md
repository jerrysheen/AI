# 任务：实现抖音分享链接视频下载模块

## 任务目标
实现一个可复用的抖音视频下载能力，能够接收一个抖音分享链接（或包含分享链接的文本），解析出对应视频，并将视频下载到本地文件，供后续 skills 继续处理。

本任务的第一目标是交付“真实可运行、可验证”的下载模块。  
如果实现结构合适，再整理为标准 skill 形式；不要为了包装成 skill 而牺牲可运行性。

## 执行要求

### 1. 先做任务分析，不要直接写最终代码
先判断并输出：
- 该任务可能的实现方式有哪些
- 当前环境下最推荐优先验证的方案是什么
- 为什么这样排序
- 每种方案的最小可行验证应该怎么做

### 2. 根据方案类型设计最小验证
不要机械地统一做相同动作，应根据方案类型设计验证方法，例如：
- HTTP 解析方案：验证短链跳转、页面可访问性、是否能提取关键字段、是否能拿到视频地址
- 第三方库方案：验证安装、导入、真实解析是否成功
- 浏览器自动化方案：验证页面能否加载、是否能获取目标视频地址、是否可下载

### 3. 找到满足验收的稳定方案后，停止继续横向扩展
不要在已有可用方案后继续把所有备选都试一遍。  
只有在当前方案无法满足验收，或稳定性明显不足时，才切换下一方案。

### 4. 每完成一个阶段都要更新进度
每个阶段结束后：
- 在终端打印：`[✓] 阶段名`
- 更新 `.task_progress.json`

`.task_progress.json` 至少记录：
- 当前阶段
- 当前任务目标
- 已尝试方案
- 当前采用方案
- 遇到的问题
- 已完成步骤
- 验收命令
- 验收是否通过
- 下一步动作

### 5. 遇到依赖和网络问题时优先自行解决
- 遇到库缺失，可直接安装
- 遇到方案跑不通，先分析原因，再决定是否切换方案
- 读取已有环境变量中的代理配置（如 HTTP_PROXY / HTTPS_PROXY），不要自行猜测代理地址
- 不要无限重试同一失败方案

### 6. 输入处理要求
输入 `input_text_or_url` 既可能是：
- 单独的抖音分享链接
- 包含分享链接的整段分享文本
- 带有多余文案、空格、换行、复制痕迹的内容

你需要先做输入归一化，提取出有效抖音链接，再继续后续解析和下载。

### 7. 下载要求
- 必须真正下载视频文件到本地
- 必须返回本地文件路径
- 必须验证本地文件实际存在
- 必须记录文件大小
- 文件命名应尽量稳定、可追踪，避免简单随机命名导致后续 skills 不易接入
- 如果下载目录不存在，应自动创建

### 8. 为后续 skills 预留可用结果
本任务结果不是“打印一个地址”，而是“产出一个本地文件”。  
因此最终结果中至少要有：
- 原始输入链接
- 解析后的最终链接
- 本地文件路径
- 文件大小
- 可选元数据（标题、作者、视频ID等）

### 9. 禁止伪造成功
- 不要使用 mock 数据冒充真实结果
- 不要在没有运行测试时声称完成
- 不要把“只解析出链接但没下载文件”当作任务完成
- 不要因为实现了函数接口就默认任务完成
- 不要把空文件、损坏文件、0字节文件当作成功下载
