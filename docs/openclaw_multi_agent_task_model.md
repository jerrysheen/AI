# OpenClaw 多 Agent 任务模型

## 认知演进与问题澄清

这份文档不只记录结论，也记录这次讨论中认知是如何逐步收敛的。因为如果直接从术语和定义讲起，很容易出现“每个概念都懂一点，但整体系统还是串不起来”的情况。

本次讨论的核心困难，不在于单个概念本身，而在于几个概念一开始很容易被混在一起：

- OpenClaw 内置能力是什么
- agent 的角色定义是什么
- task 到底是什么
- `GOALS.md / STATE.yaml / DECISIONS.md` 这些文件是干什么的
- subagent 和 skill 是什么关系
- 多个 task 并行时状态到底归谁管理

我们一开始面对的不是“怎么实现一个系统”，而是“系统到底靠什么被驱动起来”。

### 第一阶段：先明确 OpenClaw 里到底要做什么

一开始的直觉问题是：

- 如果 OpenClaw 只是创建几个 agent，那这些 agent 为什么会开始协作
- 配了几个文件以后，系统是不是就会自动运转
- 所谓多 agent，到底是在做一个复杂平台，还是只是把任务拆给几个角色

这一步最重要的澄清是：

OpenClaw 本身不会因为“有多个 agent”就自动形成协作系统。

它提供的是运行时能力，例如：

- 能启动一个新的子会话
- 能给已有会话发消息
- 能查看 session 和 subagent 状态

但“谁来拆任务、什么时候派发、派发后结果写到哪里、下一步怎么继续”，这些都不是 OpenClaw 自动给出的，而是需要我们自己定义。

因此，第一层认知收敛是：

`OpenClaw 提供的是编排能力，不是现成的协作逻辑。`

这也直接把“第一版应该做什么”缩小成一个更清晰的问题：

不是先做一个复杂的自治多 agent 平台，而是先做一个最小协作闭环。

### 第二阶段：为什么几个状态文件看起来不像“驱动器”

当系统最小闭环被提出后，新的困惑出现了：

- `GOALS.md`
- `STATE.yaml`
- `DECISIONS.md`

这些看起来更像记录文件，而不像引擎。那它们到底是不是自动驱动系统的核心？

这里的澄清非常关键：

这些文件本身不是驱动器，它们只是共享状态载体。

也就是说：

- 它们不会自己触发任务
- 不会自己唤醒 agent
- 不会自己决定下一步

真正驱动系统继续向前的是：

- `lead agent` 的行为规则
- OpenClaw 提供的 `sessions_spawn`、`sessions_send` 等内置能力

这些共享文件的作用是：

- 让当前任务可恢复
- 让协作状态可共享
- 让任务在多轮推进后仍然有一致的上下文

这一步之后，系统的因果关系才逐渐清晰起来：

不是“文件驱动 agent”，而是“agent 根据规则读取文件、调用工具、再更新文件”。

### 第三阶段：subagent 到底是不是 skill

在理解了“文件不是驱动器”之后，下一个自然问题是：

- subagent 是不是可以理解成一种 skill
- lead agent 是不是通过调用各种 subagent 来迭代自己的状态
- 不同专业 agent 是否只是共享一些底层能力

这一步的澄清结果是：

subagent 和 skill 不是一回事。

更准确地说：

- subagent 是一个会话级的 agent 实例
- skill 更像一种方法、一套说明、一类操作模式

因此，`行业研究 agent` 和 `代码主管 agent` 可以都使用“搜索”“PDF 阅读”“提炼摘要”这些底层能力，但它们并不是因为共享 skill 就共享状态。它们只是在完成各自任务时，复用了相同的底层能力。

这一步让系统结构从“能力”与“角色”中分离出来：

- 角色决定它为什么做事
- 能力决定它怎样做事

### 第四阶段：状态到底归谁

在角色和能力分开之后，最重要的问题出现了：

- 如果有多个 task 并行，状态到底归 lead 管，还是归 subagent 管
- subagent 自己有短期记忆，那这些记忆是不是也要写进共享状态
- 一个 task 切换回来时，到底应该恢复什么

这一轮讨论最后收敛出的核心结论是：

`状态的第一归属单位不是 lead，也不是 subagent，而是 task。`

这是整套模型真正稳定下来的关键点。

因为只要状态归属于 task，很多之前模糊的问题就同时被解开了：

- 一个 task 是否独立：是，因为它有自己完整的上下文
- 多个 task 是否会互相污染：不会，因为每个 task 都有独立状态
- lead 在做什么：lead 只是读取和推进某个 task
- subagent 在做什么：subagent 只是为某个 task 执行一个子任务

换句话说，agent 不是状态的“拥有者”，而是状态的“处理者”。

### 第五阶段：task 状态与长期记忆要分开

当“状态归 task”确定之后，新的问题就变成了：

- 那这些 task 状态是不是长期记忆
- 定时任务每天跑一次，是不是会一直累积在同一份状态里
- 行业知识、用户偏好、项目背景又应该放在哪

这一步的认知演进是：

`task 状态` 和 `长期记忆` 必须分层。

其中：

- `GOALS.md / STATE.yaml / DECISIONS.md` 属于 task 级工作状态
- 用户偏好、行业知识、项目背景、可复用模式属于长期记忆

task 状态的特点是：

- 围绕一个具体任务存在
- 会被不断更新
- 任务完成后应归档

长期记忆的特点是：

- 跨任务可复用
- 不应被一次任务过程污染
- 应被多个 task 在需要时读取

到这一步之后，我们对“每次起任务，到底是在起什么”也就有了统一答案：

每次起的不是一个全局 agent 状态，而是一个新的 task 单元。

### 第六阶段：如何理解多 task 切换

最后一个关键疑惑是：

- 如果同时有五个 task，lead agent 是怎么在它们之间切换的
- subagent 自己的短期记忆是否也要一起切换
- 回到某个 task 时，应该恢复的是整段会话，还是恢复任务文件

最终得到的理解是：

当存在多个 task 时，真正被切换的是一整套 task 上下文。

也就是：

- 切到某个 task
- 读取这个 task 的 `GOALS.md`、`STATE.yaml`、`DECISIONS.md` 和产出文件
- 理解当前做到哪里
- 决定是否继续调用 subagent
- 更新同一个 task 的状态

至于 subagent，它当然会有自己的短期运行时上下文，但这些上下文不必全部结构化持久化。真正需要写回 task 的，只应该是对任务推进有价值的结果摘要、产出位置、阻塞点和下一步建议。

因此，最终形成的整体认识是：

- OpenClaw 负责提供多 session / subagent 的运行时能力
- 我们负责定义角色、调度规则和状态分层
- task 是状态归属和协作推进的第一单位
- subagent 有自己的短期工作上下文，但 task 才是需要长期可恢复的对象

## 目的

本文用于总结本次关于 OpenClaw 多 Agent 协作模型讨论的关键结论。

目标是澄清以下问题：

- 在 OpenClaw 中第一步到底应该做什么
- `lead agent`、`subagent`、`task` 之间是什么关系
- 哪些内容属于任务状态，哪些内容属于长期记忆
- 多任务切换时如何避免状态互相污染

## 核心结论

第一版不应该直接做成一个庞大的自治多 Agent 平台。

更合适的做法是先实现一个最小协作闭环：

- 一个 `lead agent` 接收用户请求
- `lead agent` 读取当前任务上下文
- `lead agent` 判断是否需要分派给一个或多个 `subagent`
- `subagent` 执行边界清晰的子任务
- 子任务结果写回当前任务状态
- `lead agent` 再读取更新后的状态并产出最终结果

一句话总结：

`OpenClaw 提供会话编排能力，我们自己定义协作规则与状态模型。`

## OpenClaw 已提供什么

OpenClaw 已经内置了 session 和 sub-agent 的编排能力。

本次讨论中提到的典型能力包括：

- `sessions_spawn`：启动一个隔离的子 Agent 会话
- `sessions_send`：向已有会话发送消息
- `sessions_list`：查看当前有哪些会话
- `sessions_history`：查看某个会话的历史记录
- `subagents`：查看或管理当前 session 拉起的子 Agent

这些都是 OpenClaw 的运行时内置能力。

## 需要我们自己定义什么

OpenClaw 不会因为“已经创建了多个 agent”就自动形成协作系统。

我们仍然需要自己定义：

- agent 角色分工
- 任务分派规则
- task 的边界
- 状态持久化方式
- 子任务完成后应该回写哪些信息

只有这些规则明确后，系统运行时才会表现出“自动协作”。

## 第一版建议的角色

建议第一版只保留三个角色：

- `lead`
- `research`
- `executor`

角色说明：

- `lead`：接收用户请求、读取任务状态、决定下一步、分派子任务、汇总最终输出
- `research`：查资料、读文档、提炼结论、输出研究结果
- `executor`：执行具体实现或操作动作、输出执行结果

后续可以增加更多 specialist，但不建议放进第一版。

## Task 是第一归属单位

本次讨论中最重要的建模结论是：

`状态属于 task，不属于 lead agent，也不属于 subagent。`

每个 task 都应该被看作一个独立的工作单元，并拥有自己的持久化上下文。

一个 task 至少应包含：

- 一个 task id
- 一个负责协调它的 lead session
- 零个或多个 sub-agent session
- 一套独立的任务状态文件
- 一组独立产出

这意味着，当系统从一个 task 切换到另一个 task 时，应该加载新的 task 上下文，而不是把多个任务混在同一个全局状态里。

## Task 状态文件

本次讨论中达成一致的最小共享任务状态包括：

- `GOALS.md`：当前目标与验收标准
- `STATE.yaml`：任务拆解、当前进度、负责人、下一步动作
- `DECISIONS.md`：关键决策及其原因

这些文件不是长期记忆。

它们应该被看作：

- task 级工作记忆
- 可恢复的协作状态
- 单个任务单元的短期持久化上下文

## Task 状态不是长期记忆

本次讨论明确确认：

- `GOALS.md`
- `STATE.yaml`
- `DECISIONS.md`

都不应被视为永久记忆。

它们应该是：

- 按 task 创建
- 随 task 过程不断更新
- 在 task 完成后归档

这一点同样适用于定时任务：

- 每次定时触发通常都应该生成一个新的 task 单元
- 可复用知识应从长期记忆层读取
- 昨天的 task 状态不应直接作为今天的活动状态继续复用

## 长期记忆应该独立存放

长期记忆必须与 task 状态分层，不应该混在一起。

长期记忆的典型内容包括：

- 用户偏好
- 行业知识
- 项目背景
- 可复用流程或模板

这样设计的好处是：

- 避免不同 task 之间相互污染
- 让每个 task 可以独立恢复
- 让任务过程更容易审计和回放

## Subagent 不是 Skill

本次讨论中也明确区分了 `subagent` 和 `skill`。

`subagent` 更准确的定义是：

- 一个会话级 agent 实例
- 能被 `lead` 启动或发送消息
- 负责执行一个边界清晰的子任务

`skill` 更接近于：

- 一种方法
- 一组操作规范
- 一类工作模式

不同 agent 可以共享相同的底层能力，例如：

- 搜索
- PDF 阅读
- 摘要提炼
- 代码执行

但共享底层能力不代表共享 task 状态。

## Subagent 应该回写什么

Subagent 在运行过程中会自然拥有自己的短期运行时上下文。

这些短期上下文可能包括：

- 最近收到的消息
- 工具调用结果
- 中间推理上下文

但并不是所有内容都应该被持久化到 task 里。

真正应该回写到 task 的，是对任务推进有价值的状态，例如：

- 完成状态
- 已完成工作的摘要
- 输出文件位置
- 当前阻塞点或风险
- 建议的下一步动作

因此可以总结为：

- subagent 确实有自己的短期工作记忆
- 但写回 task 的应该只是有用的结果摘要和可交付信息

## Task 切换模型

如果系统同时存在多个 task，`lead agent` 不应该把它们当作一个全局上下文来处理。

更合理的 task 切换方式是：

1. 打开一个具体 task
2. 读取这个 task 的 `GOALS.md`、`STATE.yaml`、`DECISIONS.md` 和 outputs
3. 判断当前做到哪里、还剩什么没完成
4. 如有需要，调用对应的 subagent
5. 把结果更新回同一个 task 的状态
6. 离开该 task

这种方式能够避免状态污染，并支持多个 task 并行存在。

## 建议的三层状态模型

本次讨论最终收敛到一个三层模型：

### 1. 长期层

跨 task 共享。

例如：

- 用户偏好
- 行业事实
- 项目知识
- 可复用模板

### 2. 任务层

只属于单个 task。

例如：

- `GOALS.md`
- `STATE.yaml`
- `DECISIONS.md`
- task outputs

### 3. 会话层

只存在于运行中的 agent session 内的临时上下文。

例如：

- 当前消息历史
- 临时工具结果
- 短期推理上下文

这个分层模型是避免 task 之间相互污染的关键。

## 推荐的目录方向

本次讨论隐含的推荐目录结构大致如下：

```text
memory/
  user_profile.md
  domain_knowledge/
  reusable_patterns/

tasks/
  task_2026_04_07_001/
    GOALS.md
    STATE.yaml
    DECISIONS.md
    outputs/
      research_summary.md
      execution_notes.md
    agents/
      research/
        notes.md
      executor/
        notes.md
```

说明：

- `tasks/` 用于存放 task 级短期持久化状态
- `memory/` 用于存放长期复用记忆
- `agents/.../notes.md` 是可选项，只有在某个子任务需要可恢复的私有笔记时才需要

## 最小执行闭环

当前讨论下的第一版最小执行闭环是：

1. 用户提交请求
2. `lead` 打开当前 task 上下文
3. `lead` 判断是否要派发给 `research` 或 `executor`
4. subagent 执行子任务
5. subagent 回写任务相关结果
6. `lead` 更新 `STATE.yaml`
7. `lead` 决定继续派发下一步，或者直接返回最终答案

这是目前最简单且可落地的协作模型。

## 本次讨论确认的关键决策

- 第一版应从最小的 `lead + research + executor` 模式开始
- OpenClaw 的 session tools 是内置运行时能力，不是需要先自建的功能
- task 状态必须按 task 隔离，不能跨 task 混用
- `GOALS.md`、`STATE.yaml`、`DECISIONS.md` 是 task 工作状态，不是长期记忆
- 长期复用知识应放在独立的 memory 层
- subagent 有自己的短期上下文，但持久化回 task 的应当只是任务相关结果
- 状态的第一归属单位应该是 task，而不是具体某个 agent 实例

## 对后续实现的直接启发

如果下一步从讨论转向实现，优先应该落地的是：

- 明确 task 目录结构
- 定义 `STATE.yaml` 的最小 schema
- 定义 `lead`、`research`、`executor` 的操作规则
- 定义 subagent 回写 task 的标准格式
